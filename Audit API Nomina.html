02 — CONSTATS

Ce que j'ai trouvé

Quinze observations classées par sévérité. Clique pour déplier le détail, le diagnostic et un correctif type.

Critique

Pas de rate limit sur /generate-pack (OpenAI)

routes/GeneratePackRoutes.ts · controllers/GeneratePackController.ts
Une route publique qui appelle GPT-4o-mini sans plafond : un script de 5 lignes peut vider ta clé OpenAI en quelques minutes. Le coût est facturé à toi, pas à l'appelant.

Diagnostic

Aucun middleware de limitation. La route est ouverte (pas de requireAuth). Le contrôleur cap les counts par section (5–20), mais rien n'empêche d'envoyer 1000 requêtes/seconde avec count=20.

Correctif

Installer express-rate-limit, créer une limite stricte sur /generate-pack (ex. 10 req / 10 min / IP), et une limite globale plus souple sur le reste.

// src/middleware/rateLimit.ts import rateLimit from "express-rate-limit"; export const aiLimiter = rateLimit({ windowMs: 10 * 60 * 1000, max: 10, standardHeaders: true, message: { error: "Trop de requêtes IA, réessaie dans 10 min" }, }); // routes/GeneratePackRoutes.ts router.post("/", aiLimiter, generatePack);

Bonus

  • Logger le coût estimé par requête (input/output tokens × tarif).
  • Exiger l'auth Clerk pour le pack IA : suit l'utilisateur, pas l'IP.
Effort~2h
Critique

Aucun gestionnaire d'erreurs centralisé

22 contrôleurs · ~140 blocs try/catch identiques
Chaque endpoint répète catch (error) { res.status(500).json({ error: 'Erreur serveur' }) }. Une stack trace, un code Prisma exotique, une assertion Zod manquée — tout finit en 500 opaque côté client.

Diagnostic

L'absence de middleware errorHandler(err, req, res, next) rend impossible : la corrélation d'erreurs, le logging structuré uniforme, la conversion automatique des erreurs Prisma (P2002 → 409, P2025 → 404), et la sérialisation des ZodError.

Correctif

// src/middleware/error.middleware.ts export const asyncHandler = (fn) => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next); export function errorHandler(err, req, res, next) { if (err instanceof ZodError) { return res.status(400).json({ error: "Validation", issues: err.issues }); } if (isKnownPrismaError(err, "P2002")) return res.status(409).json(...); if (isKnownPrismaError(err, "P2025")) return res.status(404).json(...); logger.error("Unhandled", { err, reqId: req.id, path: req.path }); res.status(500).json({ error: "Erreur serveur", reqId: req.id }); }

Les contrôleurs deviennent linéaires : plus de try/catch, on throw et le middleware s'en occupe.

Gain

  • −300 lignes de boilerplate environ.
  • Format d'erreur enfin uniforme côté client.
  • Un seul endroit où ajouter Sentry / NewRelic / etc.
Effort~4h
Critique

Sécurité HTTP quasi-absente (pas de helmet)

src/server.ts · src/app.ts
Le middleware securityHeaders ne pose que X-Frame-Options: DENY (et même pas correctement : il oublie next()). Pas de CSP, pas de HSTS, pas de X-Content-Type-Options, pas de body size limit explicite.

Bug bonus

Ton middleware actuel ne fait jamais next(), donc soit il bloque tout (en réalité Express 5 timeout), soit Express l'ignore parce qu'il est branché après listen(). Lis attentivement server.ts.

// server.ts function securityHeaders(req, res, next) { res.setHeader("X-Frame-Options", "DENY"); // pas de next() ! } app.use(securityHeaders); // après app.listen — never runs // app.ts import helmet from "helmet"; app.use(helmet()); app.use(express.json({ limit: "100kb" }));

À ajouter

  • helmet() avec CSP minimale.
  • express.json({ limit: "100kb" }) pour bloquer les bodies gigantesques.
  • Limite multer sur tous les uploads (déjà fait sur cultures, à généraliser).
Effort~1h
Important

Validation Zod inline, copiée 20 fois

controllers/GenerateController.ts · etc.
Chaque contrôleur déclare son schéma, parse req.query, et fait son propre if (!parsed.success) return 400. Sortez ça dans un middleware validate(schema, "query" | "body" | "params").

Bénéfice

  • Les schémas deviennent réutilisables et exportables (utile pour un futur OpenAPI).
  • Les contrôleurs reçoivent un req.validated typé.
  • L'erreur 400 est uniforme, gérée par errorHandler.
Effort~3h
Important

Pas de pagination sur les listes

getCultures, getCategories, getNomPersonnages, …
GET /cultures renvoie toutes les cultures. À 500 enregistrements ça passe ; à 50 000, le payload tue le client mobile et la base.

Pattern proposé

GET /cultures?page=1&pageSize=20&sort=name // Réponse { data: [...], page: 1, pageSize: 20, total: 142, totalPages: 8 }

Un helper paginate(prisma.culture, { where, page, pageSize }) standardise la chose pour tous les CRUD.

Effort~3h
Important

Double montage / et /api → middlewares dupliqués

app.ts · mountRoutes("") + mountRoutes("/api")
Chaque requête passe par les routeurs deux fois, et tu maintiens 22 alias legacy en plus. C'est une dette de migration, pas un design.

Plan

  • Versionne : /api/v1/* devient le canonical.
  • Sur / sans préfixe, retourne un 308 unique vers /api/v1/... au lieu de remonter tout le tree.
  • Supprime les alias /prenoms → /nomPersonnages & co. quand le frontend a migré (mets un header Deprecation en attendant).
Effort~4h
Important

GenerateController = 1529 lignes

src/controllers/GenerateController.ts
Sept handlers de génération, plus des helpers, plus des dictionnaires de mots-clés. Découper par ressource (generators/npcs.ts, generators/lieux.ts…) et factoriser les helpers communs (splitKeywords, scoreKeywordMatch, sampleWithoutReplacement) dans services/generation/.
Effort~5h
Important

Cascade DELETE écrite à la main

deleteCulture · 12 updateMany + 1 delete en transaction
Tu mets cultureId = null sur 12 tables enfants avant de supprimer. C'est exactement ce que onDelete: SetNull fait dans Prisma. Migration unique, contrôleur de 6 lignes.

Vérifier aussi

Que toutes les FKs critiques ont une stratégie explicite (Cascade, SetNull, Restrict) — sinon Postgres bloque ou supprime des données silencieusement selon le défaut Prisma.

Effort~2h
Important

/healthz ne vérifie pas la DB

app.ts:73
Le endpoint renvoie { ok: true } tant que Node tourne. Fly.io / Vercel ne sauront jamais si Prisma a perdu la connexion. Fais un prisma.$queryRaw\`SELECT 1\` avec timeout 1s.
Effort~30min
Important

Pas de documentation OpenAPI

Repo entier
63 routes, paramétrées avec 8–12 query params chacune dans certains cas. Le README liste les routes principales mais ce n'est pas un contrat exécutable. Génère un Swagger UI à partir de zod-to-openapi.

Tu réutilises les schémas Zod du middleware validate, donc la doc reste synchronisée gratuitement. Hébergement sous /api/docs.

Effort~4h
Recommandé

Couverture de tests < 5%

__tests__/app.test.ts · GeneratePackController.test.ts · OpenAiService.test.ts
Trois fichiers de tests pour 22 contrôleurs. Avec Supertest + un schéma Prisma de test (SQLite ou Postgres jetable), tu peux couvrir tous les CRUD en quelques jours.
Effort~8h
Recommandé

Pas de request-id ni de logs structurés

utils/logger.ts
Le logger maison est OK pour démarrer mais ne sait pas tracer une requête de bout en bout. Pino + pino-http + un X-Request-Id dans le middleware errorHandler te donnent ça pour 50 lignes de config.
Effort~2h
Recommandé

CORS regex Vercel trop permissive

app.ts:51 · /^https:\/\/[a-z0-9-]+\.vercel\.app$/i
N'importe quel projet sur vercel.app peut appeler ton API avec credentials. Restreins à nomina-v3-*.vercel.app ou whiteliste explicitement les previews via env var.

Recommandé

Réponses non-DTO : tu renvoies les modèles Prisma bruts

getCultureById, getCultures, …
createdAt, updatedAt, FKs internes, champs marqués nullable mais toujours présents : tout fuite côté client. Un mapper toCultureDTO() par ressource te donne un contrat stable même si la DB bouge.

Recommandé

Normalisation de genre dispersée

GenerateController, NomPersonnageController, scripts/
La logique "m" → ["M", "Masculin", "homme", …] existe en 3+ endroits avec des variantes. Centralise dans utils/genre.ts et migre la DB pour stocker une valeur canonique (les libellés sont du presentation, pas du data).

04 — KEEPERS

Ce qui tient déjà bien

Toute critique mérite son contrepoint. Voici ce que je ne toucherais pas — au contraire, des fondations à protéger pendant les refactos.

✓
Schéma Prisma propre et structuré Trois blocs réalisme (lieux, métier/orga, relations/events) avec des types paramétrables au lieu d'enums figés. Réutilisable hors fantasy.
✓
Auth Clerk avec fallback session Le double mécanisme (verifyToken + fallbackValidateSession) gère les bords (clock skew, refresh) sans laisser l'utilisateur dans le noir.
✓
Gestion des erreurs Prisma typée isKnownPrismaError(err, "P2002") est utilisé proprement : P2002 → 409, P2025 → 404, P2003 → 409. Bonne base, à centraliser.
✓
RNG seedé et reproductible services/generation/rng.ts permet aux clients de rejouer une génération avec le même seed. Rare et précieux.
✓
Fallbacks intelligents sur les mots-clés Quand un match strict ne donne rien, l'API renvoie des suggestions classées par pertinence au lieu d'un tableau vide. C'est de l'UX serveur, pas du code de stagiaire.
✓
Scripts opérationnels en place Seed, backfill, normalize, report, clean — chaque migration métier a son script versionné. Disciplines qui paient.
